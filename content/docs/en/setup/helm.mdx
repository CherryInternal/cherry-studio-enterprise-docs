## Features

- **Dual service deployment**: Deploy API and Admin simultaneously
- **API service**: Backend API based on PostgreSQL for enterprise data storage
- **Admin console**: Web-based management UI
- **Database support**: PostgreSQL with high availability and scalability
- **Independent configuration**: Each service can be enabled/disabled and configured independently
- **Auto scaling**: Supports HPA (Horizontal Pod Autoscaler)
- **Health checks**: Includes liveness and readiness probes
- **Ingress support**: Independent Ingress rules for each service

## Installation

### Prerequisites

- Kubernetes 1.16+
- Helm 3.0+
- PVC support in the cluster

### Basic Install

Choose the correct install command based on how you obtained the chart:

**If you downloaded the release:**

```bash
# Enter the extracted directory
cd cherry-studio-enterprise-helm-main

# Install chart (note the ./helm directory)
helm install cherry-studio-enterprise ./helm

# Or specify a namespace
helm install cherry-studio-enterprise ./helm --namespace cherry-studio --create-namespace
```

**If you used git clone:**

```bash
# Enter the cloned directory
cd cherry-studio-enterprise-helm

# Install chart
helm install cherry-studio-enterprise ./helm

# Or specify a namespace
helm install cherry-studio-enterprise ./helm --namespace cherry-studio --create-namespace
```

### Custom Install

Create a `values-custom.yaml` file to override defaults:

```yaml
app:
  # API service configuration
  api:
    ingress:
      enabled: true
      annotations:
        nginx.ingress.kubernetes.io/proxy-body-size: '20m'
      hosts:
        - host: api.your-domain.com
          paths:
            - path: /
              pathType: ImplementationSpecific

  # Admin console configuration
  admin:
    ingress:
      enabled: true
      annotations:
        nginx.ingress.kubernetes.io/proxy-body-size: '20m'
      hosts:
        - host: admin.your-domain.com
          paths:
            - path: /
              pathType: ImplementationSpecific

  # Storage configuration (optional)
  persistence:
    size: 20Gi

  # Environment configuration (required for production)
  env:
    API_URL: http://cherry-studio-enterprise-api.test # Change to your backend API address
    JWT_SECRET: 'your-secure-jwt-secret-at-least-32-characters-long'
    DB_HOST: database.host
    DB_PORT: 5432
    DB_USERNAME: username
    DB_PASSWORD: password
    DB_NAME: database
    DB_TYPE: postgres

  # Resource configuration (optional)
  resources:
    limits:
      cpu: 1000m
      memory: 1Gi
    requests:
      cpu: 200m
      memory: 256Mi
```

Then install with custom configuration:

```bash
helm install cherry-studio-enterprise ./helm -f values-custom.yaml
```

## Configuration Details

### Common Configuration Scenarios

| Scenario | Key configuration | Example |
| -------------- | ------------------------------- | ---------------------------------------------- |
| **Dev/Test** | Use defaults, access via port-forward | `helm install cherry-studio-enterprise ./helm` |

**⚠️ Important**:

- You must change the default JWT_SECRET in production
- You must configure your own PostgreSQL connection in production
- Manage sensitive data like database passwords with Kubernetes Secrets

You can override these settings via `app.env`, for example:

```yaml
app:
  env:
    JWT_SECRET: 'your-custom-jwt-secret'
    ADMIN_APP_NAME: 'your app name'
    DB_HOST: 'your-postgres-host'
    DB_USERNAME: 'your-username'
    DB_PASSWORD: 'your-password'
    DB_NAME: 'your-database'
```

**Note**: In production, we recommend using Kubernetes Secrets to manage sensitive information.

## Deployment Verification

After installation, verify the deployment with the following commands:

```bash
# Check pod status
kubectl get pods -l app.kubernetes.io/name=cherry-studio-enterprise

# Check service status
kubectl get svc -l app.kubernetes.io/name=cherry-studio-enterprise

# Check persistent volumes
kubectl get pvc

# Run connection test
helm test cherry-studio-enterprise
```

### Deployment Checklist

- **Pod status**: All pods are Running
- **Service status**: Services are created with endpoints
- **PVC status**: PVCs are bound to available PVs
- **Ingress status**: If enabled, Ingress rules are created
- **Health checks**: API responds to health checks
- **Data persistence**: Data persists after pod restarts
- **JWT configuration**: Default JWT_SECRET replaced in production

## Access the Application

### Use Port-Forward (Recommended for Testing)

**API service:**

```bash
kubectl port-forward svc/cherry-studio-enterprise-api 3670:3670
```

Then visit `http://localhost:3670`

**Admin console:**

```bash
kubectl port-forward svc/cherry-studio-enterprise-admin 3680:3680
```

Then visit `http://localhost:3680`

### Use LoadBalancer

If you configured a LoadBalancer service:

**API service:**

```bash
kubectl get svc cherry-studio-enterprise-api
```

Access `http://EXTERNAL-IP:3670`

**Admin console:**

```bash
kubectl get svc cherry-studio-enterprise-admin
```

Access `http://EXTERNAL-IP:3680`

### Use Ingress

If Ingress is enabled, access via the configured domain names:

- API service: via the domain set in `app.api.ingress.hosts`
- Admin console: via the domain set in `app.admin.ingress.hosts`

### Common Issues

1. **Pods stuck in Pending**

   - Check node resources
   - Ensure PVC can be created and bound

2. **Service not accessible**

   - Check Service and Endpoints
   - Verify firewall rules

   - Ensure PVC is configured correctly
   - Check storage class persistence support

## Cluster Configuration Reference

Based on organization size, the following cluster configurations are recommended:

### Recommended Specs

| User scale | Node spec | Node count | Pod replicas | Storage |
|---------|---------|---------|-----------|----------|
| **0-50 users** | 2 vCPU, 4GB RAM | 1-2 | 1 | 10-20GB |
| **50-200 users** | 4 vCPU, 8GB RAM | 2-3 | 1-2 | 20-50GB |
| **200-1000 users** | 8 vCPU, 16GB RAM | 3-5 | 2-3 | 50-100GB |
| **2000+ users** | 16 vCPU, 32GB RAM | 5+ | 3-5 | 100GB+ |

### Resource Usage Notes

Cherry Studio Enterprise is lightweight:
- **Memory**: ~100-150MB after startup
- **CPU**: < 100m under normal load
- **Database load**: very low, mostly configuration data
- **Storage**: mainly logs and temporary files

Even small clusters can support many users. Adjust resources based on actual monitoring data.

### Database Configuration

Database requirements are modest. Recommended setup:
- **Small deployments (< 200 users)**: single PostgreSQL instance (2 vCPU, 4GB RAM)
- **Large deployments (> 200 users)**: PostgreSQL primary-replica for availability rather than performance

## Notes

1. **Instance recommendation**: Although PostgreSQL is used, to ensure data consistency and app state management, run a single pod. For high availability, configure PostgreSQL primary-replica.

2. **Data persistence**: Ensure the cluster supports persistent storage; otherwise data is lost when pods restart.

3. **JWT secret security**:

   - Replace the default JWT_SECRET in production
   - Use Kubernetes Secrets for sensitive data
   - JWT secrets should be strong random strings (at least 32 characters)

4. **Network security**:
   - Use Ingress + TLS in production
   - Restrict network access
   - Update image versions regularly

### Example JWT Secret Configuration for Production

```yaml
# Use a Secret to manage the JWT secret
apiVersion: v1
kind: Secret
metadata:
  name: cherry-studio-jwt-secret
type: Opaque
data:
  jwt-secret: <base64-encoded-strong-random-string>
---
# Reference in values.yaml
app:
  env:
    JWT_SECRET:
      valueFrom:
        secretKeyRef:
          name: cherry-studio-jwt-secret
          key: jwt-secret
```
